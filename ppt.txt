## SLIDE 02 — PROBLEM STATEMENT

**Section Label:** 01 · THE PROBLEM

**Slide Title:** The MCA Placement Preparation Gap

**Content — 4 Problems in a 2×2 Grid:**

---

### Problem 01 — Fragmented Toolchain
> Students simultaneously manage WhatsApp for announcements, Google Sheets for attendance, a separate LeetCode account for practice, and the college eCampus portal for academic data. No single platform connects or reflects their actual placement readiness.

- No unified view of preparation progress
- Data scattered across 4–5 unconnected tools
- Students cannot measure what matters

---

### Problem 02 — Zero Readiness Visibility
> Neither students nor faculty have a real-time, honest signal of how prepared any student actually is for an upcoming placement drive. Faculty receive manually compiled reports weeks after the fact, making timely intervention impossible.

- No continuous performance indicator for students
- Faculty cannot identify at-risk students in time
- Preparation gaps discovered only at the placement drive itself

---

### Problem 03 — Institutional Memory Lost Every Batch
> Every batch builds valuable knowledge — interview experiences, company-specific tips, preparation strategies. This knowledge lives in personal chats and memories. When a batch graduates, it is permanently lost. The next batch starts from zero every single time.

- Interview experiences never recorded or archived
- Juniors cannot learn from seniors' real placement encounters
- The department has no growing knowledge repository

---

### Problem 04 — Manual Administrative Overhead
> The Placement Representative manually collects attendance sheets from Team Leaders, aggregates them across teams of 120 students, calculates percentages by hand, and compiles reports for faculty — consuming 3–5 hours every week that should be spent on actual placement preparation.

- Repetitive data entry and calculation errors
- No automated reporting or alerting system
- Placement Rep time wasted on administration, not strategy

---

**Speaker Notes:**
The problem has four distinct dimensions: tool fragmentation, visibility gaps, knowledge loss, and administrative burden. Every one of these is real and experienced by our batch today. PSGMX was built specifically to solve all four.

## SLIDE 03 — PROPOSED SOLUTION

**Section Label:** 01 · THE SOLUTION

**Slide Title:** PSGMX — One Ecosystem, Two Platforms, One Database

**Content:**

### The Core Idea
PSGMX is a dual-platform placement readiness ecosystem built entirely by students of the MCA Department at PSG College of Technology. Instead of replacing existing institutional tools, it fills the placement-preparation layer that currently does not exist.

### The Two-Platform Architecture

---

**Platform 01 — PSGMX Mobile App**
**URL: app.psgmx.tech**

The daily habit tool. Students open it every morning — the same way they open a fitness or language app — to track their placement preparation through small, consistent daily actions.

| Feature | Description |
|---|---|
| Daily Five Quiz | 5 MCQs/day, mixed topics, streak-based |
| Quest Board | Daily LeetCode + core-subject task |
| Placement Attendance | Session tracking, team-scoped marking |
| LeetCode Leaderboard | Batch-scoped ranking, 6-hour sync |
| Placement Log | Company records + interview experiences |
| AI Mentor Chat | OpenRouter-powered, grounded in Knowledge Brain |
| Readiness Score | Live 0–100 score, updated in real time |

**Distribution:** Android APK via GitHub Releases · iOS via PWA (Safari, no App Store needed)

---

**Platform 02 — PSGMX Web Platform**
**URL: psgmx.tech**

The knowledge and examination tool. Students, faculty, and alumni use this when they need depth — reading interview guides, sitting proctored mock exams, contributing knowledge, or reviewing student progress.

| Feature | Description |
|---|---|
| Knowledge Brain | Searchable articles and interview guides |
| AI Senior Chat | RAG-powered AI using real department knowledge |
| Mock Examinations | Proctored exams (camera + tab detection) |
| Alumni Lineage Network | Senior–junior matching by roll number |
| Faculty Mentee Roster | Real-time readiness scores for all mentees |
| HOD Department View | Batch-level aggregate analytics |

---

### The Unifying Principle
**One Supabase PostgreSQL database** — shared by both platforms. No sync jobs. No bridge APIs. When a student completes their Daily Five on the mobile app, the readiness score updates, and the faculty member on the web platform sees it within 60 seconds. The **roll number** is the universal identity key that connects everything.

---

**Speaker Notes:**
The key insight is that these two platforms serve different moments in a student's day. The mobile app is opened by habit — every morning. The web platform is opened with intent — when there's a specific need. Because they share one database, data flows naturally between them without any integration complexity.

## SLIDE 04 — OBJECTIVES OF THE PROJECT

**Section Label:** 02 · PROJECT GOALS

**Slide Title:** Objectives of the Proposed System

**Content — 7 Objectives:**

---

**01. Build a unified daily habit platform for placement preparation**
Design and develop a mobile-first application that encourages consistent daily engagement through streaks, tasks, and leaderboards, covering every aspect of placement readiness in one interface.

**02. Deliver a real-time Readiness Score computed from verifiable activity**
Replace subjective self-assessment with an objective, continuously updated 0–100 score derived from four measurable inputs: Daily Five engagement, LeetCode progress, mock exam performance, and placement attendance.

**03. Automate the full batch lifecycle without manual intervention**
Implement a system where new students enroll automatically by roll number, graduating students transition to alumni status programmatically, and each new batch inherits the knowledge accumulated by all previous batches.

**04. Build a searchable, AI-augmented institutional Knowledge Brain**
Create a growing repository of interview experiences, preparation guides, and placement insights contributed by students and alumni — indexed with vector embeddings and accessible through an AI-powered chat interface.

**05. Conduct and proctor mock examinations at placement-grade rigor**
Provide faculty with tools to create, schedule, and administer proctored mock examinations — with camera monitoring, tab-switch detection, and per-student question randomization — whose scores feed directly into the Readiness Score.

**06. Connect current students to alumni through a lineage network**
Automatically link each student to the alumnus whose roll number matches theirs by suffix pattern, enabling direct mentorship connections, knowledge handover, and career guidance.

**07. Give faculty real-time, zero-effort visibility into student readiness**
Ensure that every faculty member can see the current readiness score, component breakdown, and 90-day trend for every assigned mentee — without manually collecting any data.

---

**Speaker Notes:**
Note that objective 3 — the batch lifecycle automation — is what makes this system sustainable beyond our batch. By June 2027, when we graduate, the system transitions us to alumni automatically, the next batch takes our place, and the system continues running without anyone touching the code.

**Slide Title:** Functional Requirements

---

### PSGMX MOBILE APP — app.psgmx.tech

- FR07: Completing all 5 questions increments current streak by 1
- FR08: Missing a day resets streak to 0 unless a freeze token is consumed
- FR09: System grants 2 freeze tokens per student per calendar month; unused tokens do not carry over

**Quests — Daily Task Management**
- FR10: Coordinators and Placement Reps publish one LeetCode task + one core-subject task per day per batch
- FR11: Students mark tasks complete; Team Leaders verify completions
- FR12: Task completion feeds into the Readiness Score calculation

**Sessions — Placement Attendance**
- FR13: Coordinators/Reps schedule placement-prep sessions with date, topic, and target teams
- FR14: Push notifications sent automatically to eligible students on session creation
- FR15: Team Leaders mark present/absent/excused for each member of their team per session
- FR16: Placement-session attendance percentage computed automatically and shown to students

**Placement Log**
- FR17: Placement Reps log company visit records (name, date, roles, package band, rounds)
- FR18: Second-year students add personal experience writeups per round per company
- FR19: Approved writeups auto-ingest into the Next.js Knowledge Brain via database trigger
- FR20: All placement logs remain permanently visible to future batches after graduation

**LeetCode Integration**
- FR21: LeetCode stats synced every 6 hours via public API
- FR22: Batch-period weighted score computed: Easy×1 + Medium×2 + Hard×3
- FR23: Batch-scoped percentile rank computed and displayed; two separate leaderboards (Readiness + LeetCode)

**AI Mentor Chat**
- FR24: AI Mentor powered by OpenRouter (free-tier models with fallback chain)
- FR25: Before sending to AI model, query Next.js Knowledge Brain for relevant context
- FR26: If AI unavailable, return pre-written contextual fallback — never an error screen

---

### PSGMX WEB PLATFORM — psgmx.tech

**Mock Examinations (Faculty)**
- FR27: Faculty create mock exams with configurable question types, duration, and proctoring level
- FR28: Proctoring: camera access required; tab-switch detection; per-student question randomization
- FR29: Exam results written to Supabase immediately on submission; Readiness Score updates within 60 seconds

**Knowledge Brain**
- FR30: Students and alumni submit articles; faculty review and approve before indexing
- FR31: Approved articles chunked and embedded using Supabase AI (gte-small) into pgvector index
- FR32: Semantic search across all approved articles using cosine similarity

**AI Senior Chat (RAG)**
- FR33: Student question generates an embedding; top 3 matching Knowledge Brain chunks retrieved
- FR34: Chunks passed as context to Gemini 2.5 Flash (primary) or OpenRouter Claude (fallback)
- FR35: If all AI calls fail, return a static fallback — never an error message

**Alumni & Lineage**
- FR36: On student creation, lineage_map auto-populated by matching roll number suffix across batches
- FR37: Alumni toggle mentorship availability; matched juniors see "Your Senior" card on both platforms
- FR38: Alumni contribute to Knowledge Brain and Collaboration Marketplace

**Faculty Mentee Roster**
- FR39: Faculty see all assigned mentees' current Readiness Score, 4-component breakdown, and 90-day trend
- FR40: Students with band "At Risk" (0–39) or declining trend for 14+ days flagged automatically

**HOD Dashboard**
- FR41: HOD sees batch-level average scores, session attendance trend, drives completed, company packages
- FR42: HOD access is read-only; no write permissions on student data

**Batch Graduation (Automated)**
- FR43: Supabase CRON Edge Function runs on configured graduation date per batch
- FR44: Batch status → graduated; student roles → alumni; Flutter app → read-only archive
- FR45: Graduation screen shown once to each student summarizing their 2-year journey

---

**Speaker Notes:**
The key functional distinction between the two platforms: the mobile app handles everything that benefits from daily mobile-first access. The web platform handles everything that benefits from a full desktop screen — reading long articles, sitting a 60-minute exam, or reviewing 120 students' data in a faculty dashboard.

- Supports 240 active students across two batches simultaneously

- 99.9% database uptime guaranteed by Supabase service-level agreement
- Offline-first local cache on Android (Drift/SQLite) ensures Daily Five and readiness data accessible without connectivity

- Row Level Security (RLS) enforced on every Supabase table — no client can access data outside its permitted scope
- Service Role Key (admin access) never included in any client-side or mobile code

### Privacy
- Individual Daily Five question responses never written to any database
- Individual readiness score components visible only to the student and their assigned faculty mentor

- Mock exam answers stored only as aggregate score; individual responses not retained

### Usability
- Zero manual enrollment — roll number parsing eliminates the need for any whitelist or seed file
- Placement Representative can operate the entire system with basic smartphone familiarity
- Batch handover on graduation requires no developer involvement — fully automated

### Maintainability
- Entire system maintainable by one non-technical student Placement Rep with no DevOps knowledge
- New batch joins automatically; old batch graduates automatically; code does not change

- Android users: APK downloadable from GitHub Releases at any time (no Play Store dependency)
- iOS users: PWA installable via Safari — push notifications and offline cache supported
- Web platform deployed on Vercel — zero-downtime deployments via GitHub Actions on every push to main

### Data Integrity
- All 20+ Supabase tables have RLS enabled — no unauthorized writes possible
- Foreign key constraints with CASCADE deletes prevent orphaned records
- Audit log table records every sensitive action (role changes, graduation triggers, approvals)
- Readiness Score computed server-side by Edge Function — cannot be manipulated by client

---

**Speaker Notes:**
The privacy requirement about Daily Five answers is worth explaining carefully. We deliberately do not store individual question responses. The system computes a running accuracy rate at the moment of submission and discards everything else. This means we cannot recreate what any student answered on any given day — which is exactly the design intent.

**Architecture Highlights:**
- **Single database** eliminates sync complexity between the two apps
- **RLS on every table** means both apps safely share the same database without risk
- **Edge Functions** run server-side Deno code — score computation, graduation, and knowledge ingestion are all automated triggers
- **Realtime engine** broadcasts readiness score changes to Flutter app within seconds of a mock exam submission on the web platform

---

**Speaker Notes:**
The single-database decision is the most important architectural choice in this project. When I first designed this, I considered two databases with a bridge API. But that adds latency, sync errors, and maintenance burden. One database with proper RLS is simpler, faster, and more reliable. The trade-off is that both apps must connect to the same Supabase project — which is fine since they serve the same institution.

## SLIDE 13 — MATCHING ALGORITHM / READINESS SCORE LOGIC

**Section Label:** 06 · CORE ALGORITHM

**Slide Title:** Readiness Score Engine — The Algorithm

**Content — Formula Breakdown:**

---

### The Four Inputs

---

**INPUT 01 — Daily Five Engagement**
**Maximum contribution: 30 points**

```
daily_five_score = (Adherence Rate × 20) + (Accuracy Rate × 10)

Where:
  Adherence Rate   = days all 5 questions completed ÷ total eligible days (last 90)
  Accuracy Rate    = total correct answers ÷ total questions answered (running rate)
  Streak Bonus     = up to +5% on adherence for active streaks ≥ 30 days

Example: 80% adherence + 70% accuracy = (0.80 × 20) + (0.70 × 10) = 16 + 7 = 23/30
```

---

**INPUT 02 — LeetCode Progress (Batch Period)**
**Maximum contribution: 25 points**

```
leetcode_score = batch_percentile × 0.25

Where:
  batch_weighted_score = (Easy solved × 1) + (Medium × 2) + (Hard × 3)
  [Only problems solved DURING the batch period — June 2025 onwards]
  batch_percentile     = rank within own batch (0–100)
  [Always compared within same batch — fair for both 1st and 2nd years]

Example: 85th percentile in batch = 0.85 × 25 = 21.25/25
```

---

**INPUT 03 — Mock Exam Performance**
**Maximum contribution: 35 points**

```
mock_exam_score = decay_weighted_average × 0.35

Decay weights by recency:
  Within last 30 days  → weight = 1.00 (full weight)
  30 to 90 days ago    → weight = 0.70
  Older than 90 days   → weight = 0.40

Example: One recent exam at 80%, one older at 60%
  = ((80 × 1.0) + (60 × 0.40)) ÷ (1.0 + 0.40) = (80 + 24) ÷ 1.40 = 74.3
  = 74.3 × 0.35 = 26.0/35

NOTE: Student with ZERO mock exams scores 0/35 — strongest incentive to sit exams early.
```

---

**INPUT 04 — Placement Session Attendance**
**Maximum contribution: 10 points**

```
session_score = (sessions_attended ÷ sessions_eligible) × 10

Where:
  sessions_attended  = placement sessions where student was marked present or excused
  sessions_eligible  = all sessions targeting this student's batch or team
  [If no sessions scheduled yet, this input = 0 (not penalized)]
```

---

### The Final Formula

```
┌────────────────────────────────────────────────────────────────────────┐
│                                                                        │
│   READINESS SCORE  =  D5 Score  +  LC Score  +  ME Score  +  SA Score │
│        (0–100)           (0–30)      (0–25)       (0–35)      (0–10)  │
│                                                                        │
│   Computed server-side by Supabase Edge Function.                      │
│   Never writable by client. Updated within 60 seconds of any event.   │
└────────────────────────────────────────────────────────────────────────┘
```

---

### Score Bands

| Band | Score Range | Interpretation | Faculty Action |
|---|---|---|---|
| **STRONG** | 80 – 100 | Consistently active across all four dimensions | Monitor, encourage |
| **BUILDING** | 60 – 79 | Active but likely missing one input dimension | Check which input lags |
| **NEEDS ATTENTION** | 40 – 59 | Present but inconsistent across inputs | Schedule check-in |
| **AT RISK** | 0 – 39 | Low engagement or zero mock exams taken | Immediate intervention |

---

### Roll-Suffix Lineage Matching (Alumni Algorithm)

```
Student 25MX223 signs up
  → Extract suffix "223"
  → Query users WHERE roll_no LIKE '%MX223' AND batch.status = 'graduated'
  → Find 24MX223 (graduated, role = alumni)
  → Insert lineage_map (junior_user_id=25MX223, senior_user_id=24MX223, roll_suffix="223")
  → If 24MX223 has mentorship_open = true → show "Your Senior" card in Flutter app
```

---

**Speaker Notes:**
The mock exam weight of 35 points is deliberate. It is the single highest-weighted input because a proctored exam is the closest approximation of a real placement test. A student with great streaks but no exams taken cannot score above 65/100. This creates a visible, material incentive to engage with the web platform at least once per semester.

## SLIDE 14 — KEY TECHNICAL CHALLENGES AND SOLUTIONS

**Section Label:** 07 · IMPLEMENTATION

**Slide Title:** Key Technical Challenges and How They Were Solved

**Content — 4 Challenge-Solution Pairs:**

---

### Challenge 01 — Two Apps Sharing One Database Safely

**Problem:**
Both the Flutter app and the Next.js platform must read and write to the same Supabase database. A student on the mobile app must not be able to see another student's score. A faculty member must see all students. Alumni must see only their own historical data. How do you enforce these rules at the database level without relying on application-layer checks that could be bypassed?

**Solution:**
Supabase's Row Level Security (RLS) is enabled on every table. RLS policies are written in SQL and enforced by PostgreSQL itself — they cannot be bypassed regardless of which application connects or what query it runs. The mobile app uses the anon key (minimal access, governed by RLS). The web platform uses the service role key only in server-side API routes, never in browser code. The rule is: the client can only read and write its own permitted rows.

---

### Challenge 02 — Batch Lifecycle Without Manual Intervention

**Problem:**
A graduating batch of 120 students must have their accounts transitioned to alumni status, their Flutter app converted to read-only archive, and the next junior batch promoted to senior — all on a specific date in June, without a developer doing anything, without disrupting active users, and without corrupting any data.

**Solution:**
A Supabase CRON Edge Function (`batch-graduation`) runs on a configured date (e.g., `0 0 1 6 *` — midnight June 1st). It executes a single database transaction: flip batch status to `graduated`, update all student roles to `alumni`, insert graduation notifications for each user, demote the junior batch to senior, and log the entire operation in `audit_logs`. If any step fails, the entire transaction rolls back — no partial states.

---

### Challenge 03 — Vector Search Without External Infrastructure

**Problem:**
The Knowledge Brain needs semantic vector search — so a student asking "What is TCS's DSA round like?" gets articles about TCS interviews, not just keyword matches on "TCS" or "DSA." Traditional vector databases (Pinecone, Weaviate) add external infrastructure, cost, and complexity. How do you add real semantic search without leaving the single-database architecture?

**Solution:**
Supabase's `pgvector` extension runs inside PostgreSQL — no external service required. Articles are chunked into 500-character segments and embedded using Supabase's native AI embedding (`gte-small` model), available directly in Edge Functions without any API key. Embeddings are stored as `vector(1536)` columns with an IVFFlat index. Semantic search is a single SQL query using the `<=>` cosine distance operator. Everything stays in one database.

---

### Challenge 04 — iOS Support Without App Store Publishing

**Problem:**
The project has no budget for Apple's $99/year developer fee and no institutional backing to publish on the App Store. But iOS users (approximately 40% of Indian MCA students) cannot sideload APKs the way Android users can. Excluding iOS entirely would leave a significant portion of the batch without access.

**Solution:**
The Flutter app is compiled as a Progressive Web App (PWA) and deployed to Firebase Hosting at `app.psgmx.tech`. An iOS user opens this URL in Safari, taps "Add to Home Screen," and from that point on the app behaves exactly like a native app — it appears on the home screen, supports web push notifications for exam reminders and streak nudges, and uses IndexedDB for offline caching of the Daily Five question set and readiness data. The only capability not available on iOS PWA is screenshot blocking during the quiz (a Safari sandbox limitation) — which is documented and accepted.

---

**Speaker Notes:**
Challenge 3 was the one that initially seemed hardest. The first instinct was to use a dedicated vector database service. But then I discovered pgvector — and realized the entire semantic search capability could live inside Supabase without adding any external dependency. That's when the "one database for everything" architecture became obviously right.

## SLIDE 15 — SECURITY AND PRIVACY MEASURES

**Section Label:** 08 · SECURITY

**Slide Title:** Security Architecture and Privacy Design

**Content — 6 Security Areas:**

---

### 01 — Authentication (Identity Verification)
- **Method:** Supabase Auth with OTP delivery to institutional email
- **Domain restriction:** Only `@psgtech.ac.in` email addresses accepted — enforced at auth layer and via database `CHECK` constraint
- **No passwords:** Students never create or remember a password; OTP is single-use and expires in 10 minutes
- **Session management:** JWT tokens managed by Supabase; refresh handled automatically by `@supabase/ssr` middleware

---

### 02 — Authorization (Access Control)
- **Row Level Security (RLS):** Enabled on all 20+ Supabase tables — no table is publicly readable or writable
- **Role-based policies:** SELECT, INSERT, UPDATE, DELETE policies written per role per table — a student cannot see another student's exam score regardless of which query they run
- **Service role isolation:** Admin-level database access (service role key) used only in Next.js server-side API routes and Edge Functions — never exposed to any browser or mobile client
- **Permission flags:** Granular per-user capabilities stored in `user_permissions` table; the Placement Rep is the only role that can grant permissions

---

### 03 — Data Privacy
- **Daily Five answers never stored:** Individual question responses exist only in memory for the duration of the quiz; only the running accuracy rate is written to the database
- **Score visibility:** Individual readiness score components visible only to the student and their assigned faculty mentor — not to peers, not to alumni
- **Mock exam privacy:** Individual question responses stored only as aggregate score; answer-level data not retained after submission
- **Alumni scope:** After graduation, alumni cannot see current students' individual scores — only their own historical record

---

### 04 — API and Transport Security
- **HTTPS everywhere:** All Supabase, Vercel, Firebase, and external API calls use TLS 1.2+ — no unencrypted communication at any layer
- **eCampus credentials:** Custom eCampus passwords stored encrypted at rest in Supabase; never included in API responses or logs
- **Edge Function security:** Internal Edge Functions called via database trigger with a pre-shared secret stored in Supabase secrets — not exposed in source code
- **Environment variables:** All keys (Supabase URL, anon key, service role key, AI keys) stored in environment configuration — no credentials in source code

---

### 05 — Examination Integrity
- **Camera access required:** Mock exam on the web platform requires camera permission before the exam begins; if denied, exam cannot start
- **Tab-switch detection:** `document.visibilitychange` event monitored throughout exam; each violation logged as a proctoring flag in `mock_exam_results.proctoring_flags`
- **Question randomization:** Each student receives a unique question ordering derived from their `user_id` as a seed — answers cannot be shared between students in the same session
- **Daily Five protection (Android):** Screenshot and screen recording blocked during the 5-question session; app-backgrounding forfeits the attempt
- **Daily Five limitation (iOS PWA):** Safari sandbox does not support screenshot blocking — documented limitation; tab-switch forfeiture still applies

---

### 06 — Database and Infrastructure Security
- **Audit trail:** `audit_logs` table records every sensitive action: role changes, permission grants, graduation triggers, article approvals, exam creation
- **Cascade constraints:** Foreign keys with `ON DELETE CASCADE` prevent orphaned records; no dangling data on user deletion
- **pgvector access:** Knowledge embeddings table accessible only through the Next.js server-side API — not directly queryable by any client
- **Supabase SLA:** 99.9% uptime guaranteed; automated daily backups; point-in-time recovery available

---

**Speaker Notes:**
The key security insight is that RLS makes the database self-defending. Even if a bug in the application code accidentally runs the wrong query, PostgreSQL enforces the access rules before returning any data. We don't rely on application-layer checks alone. This is defense in depth — multiple independent layers, so no single failure exposes the data.

## APPENDIX D — Edge Function: compute-readiness-score (Logic Summary)

```
FUNCTION: compute-readiness-score(user_id)
─────────────────────────────────────────────────────────────

INPUT: user_id (UUID)

STEP 1: Fetch daily_five_streaks for user
  adherence_rate = total_days_completed / eligible_days_last_90
  accuracy_rate  = running_accuracy_rate / 100
  daily_five_score = clamp((adherence_rate × 20) + (accuracy_rate × 10), 0, 30)

STEP 2: Fetch leetcode_stats for user
  leetcode_score = clamp(batch_percentile × 0.25, 0, 25)
  [Also recompute batch_percentile for ALL users in this batch
   if leetcode_stats was the trigger event]

STEP 3: Fetch all mock_exam_results for user
  For each result:
    age = days since submitted_at
    weight = 1.00 if age < 30, 0.70 if 30–90, 0.40 if > 90
  weighted_avg = sum(score × weight) / sum(weights) [or 0 if no results]
  mock_exam_score = clamp(weighted_avg × 0.35, 0, 35)

STEP 4: Compute session_score
  attended = COUNT(placement_attendance WHERE student_id=user_id AND status IN ('present','excused'))
  eligible = COUNT(sessions that targeted this student's batch or team)
  session_score = clamp((attended / max(eligible, 1)) × 10, 0, 10)

STEP 5: Compute total
  total = daily_five_score + leetcode_score + mock_exam_score + session_score

STEP 6: UPSERT readiness_scores
  { user_id, score=total, components, band, computed_at=now() }

STEP 7: UPSERT readiness_score_history
  { user_id, score=total, components, snapshot_date=today }
  [ON CONFLICT (user_id, snapshot_date) DO UPDATE]

OUTPUT: Updated readiness_scores row
SIDE EFFECT: Supabase Realtime broadcasts change to all subscribers
─────────────────────────────────────────────────────────────
```
